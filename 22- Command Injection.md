# Layer 1
## يعني إيه Command Injection؟

خلينا الأول نفهم يعني إيه **Command**.

الـ Command هو الأمر اللي بتكتبه في الـ Terminal أو الـ CMD أو الـ Bash.

زي:

```bash
ls
```

يعرض الملفات.

أو:

```bash
whoami
```

يعرفك أنت شغال بأي يوزر.

أو:

```bash
pwd
```

يعرفك أنت واقف في أنهي فولدر.

---

## طيب يعني إيه Injection؟

كلمة Injection معناها "حقن".

يعني البرنامج كان ناوي ينفذ حاجة معينة...

وأنت حقنت فيه كود أو أمر زيادة يخليه يعمل حاجة تانية.

يعني المطور كان عايز ينفذ:

```
touch report.pdf
```

وأنت خليته ينفذ:

```
touch report.pdf
whoami
```

أو

```
touch report.pdf && whoami
```

فبقى نفذ الأمرين.

---

## أول فقرة

> A Command Injection vulnerability is among the most critical types of vulnerabilities.

يعني:

ثغرة الـ Command Injection تعتبر من أخطر الثغرات الموجودة.

ليه؟

لأنها بتخليك تنفذ أوامر على السيرفر نفسه.

---

## يعني إيه السيرفر؟

أي موقع شغال على جهاز.

مثلاً

```
facebook.com
```

وراها جهاز Linux.

الجهاز ده هو اللي بيستقبل الريكوستات.

لو قدرت تنفذ أوامر عليه...

يبقى أنت بقيت تقريباً بتتحكم فيه.

---

## الجملة دي

> It allows us to execute system commands directly on the back-end hosting server

يعني:

تقدر تشغل أوامر Linux أو Windows مباشرة على السيرفر.

زي

```

id
hostname
```

أو على Windows

```

```

---

## الجملة دي

> which could lead to compromising the entire network

يعني:

لو السيرفر ده موجود جوه شركة...

ممكن بعد ما تدخل عليه تبدأ تتحرك لباقي الأجهزة.

وده اسمه

```
Lateral Movement
```

يعني بدل ما تسيطر على جهاز واحد...

تسيطر على الشبكة كلها.

## الخلاصة

الفكرة كلها في سطر واحد:

- الموقع بياخد Input من المستخدم.
    
- المطور يحطه داخل أمر نظام التشغيل.
    
- لو المدخلات مش متفلترة أو متأمنة، المهاجم يقدر يغيّر شكل الأمر.
    
- النتيجة ممكن تكون تنفيذ أوامر غير مقصودة على السيرفر، ودي من أخطر الثغرات لأنها قد تؤدي للسيطرة على النظام بالكامل.
    

وده الأساس اللي هتبني عليه كل طرق اكتشاف واستغلال ثغرات **OS Command Injection** بعد كده.

```

```


---

## 1) Semicolon ;

دي أسهل واحدة.

معناها ببساطة

> خلص الأمر الأول... وبعده نفذ الأمر اللي بعده.

يعني

```
Command1 ;
Command2
```

السيرفر هيعمل

```
نفذ الأول

بعده نفذ التاني
```

بغض النظر الأول نجح ولا فشل.

---

## 2) New Line

ودي ناس كتير بتنساها.

الـ Shell أصلاً بيعتبر

Enter

معناه

ابدأ أمر جديد.

يعني

```
ping
```

Enter

```
whoami
```

دي بالنسبة للـ Shell أمرين.

---

فلو قدرت تدخل

```
New Line
```

جوه الـ Input...

هيعتبر إن فيه أمر جديد.

---

## 3) علامة &

في Linux

```
&
```

بتشغل الأمر في الخلفية.

يعني

```
Command1 &
Command2
```

الاتنين يشتغلوا.

---

الكتاب كاتب ملاحظة:

غالبًا هتشوف ناتج الأمر التاني الأول.

ليه؟

لأن الأول شغال Background.

---

## 4) Pipe |

دي مختلفة شوية.

الـ Pipe الطبيعي وظيفته إنه يبعت **ناتج الأمر الأول كمدخل للأمر الثاني**.

مثلاً:

```bash
cat file.txt | grep admin
```

يعني:

- `cat` يقرأ الملف.
    
- الناتج يروح لـ `grep`.
    
- `grep` يدور على كلمة `admin`.
    

لكن في بعض تطبيقات الويب، لو قدرت تحقن `|`، ممكن تلاحظ إن **النتيجة اللي بتظهر لك هي نتيجة الأمر اللي بعد الـ Pipe فقط**، وده بيساعدك تعرف إن الحقن نجح.

---

## 5) &&

دي معناها

نفذ الأمر الثاني

**لو الأول نجح فقط.**

مثلاً

```
ping google.com && whoami
```

لو الـ Ping نجح

هيشغل

```
whoami
```

---

ولو الـ Ping فشل؟

ولا حاجة.

---

## 6) ||

دي العكس.

نفذ الأمر الثاني

لو الأول فشل.

يعني

```
ping InvalidIP || whoami
```

الـ Ping هيفشل.

إذن

هيشغل

```
whoami
```

---

## 7) Backticks ``

دي خاصة بـ Linux.

لما تكتب أمر بين علامتي ``

الـ Shell ينفذه الأول.

ودي اسمها **Command Substitution**.

يعني بدل ما تعتبر اللي جوه نص، تعتبره ناتج أمر.

---

## 8) $()

دي نفس فكرة الـ Backticks.

برضو اسمها

Command Substitution.

ودي النسخة الأحدث والأكثر استخدامًا في Bash.

---

## ليه كل الـ Operators دي مهمة؟

لأنك مش هتلاقي كل المواقع بتقبل نفس الحاجة.

ممكن موقع يمنع

```
;
```

لكن يقبل

```
&&
```

وموقع تاني يمنع

```
&&
```

لكن يقبل

```
|
```

وموقع تالت يمنع كل دول

لكن ينسى

```

```

عشان كده البنتستر بيجرب أكتر من Operator، مش واحد بس.

---

## ليه فيه URL Encoded جنب كل Operator؟

لأن أحيانًا المتصفح أو التطبيق بيمنع إرسال الرموز دي بشكل مباشر.

فبدل ما تبعت:

```
;
```

بتتبعت بالشكل المشفر الخاص بالـ URL.

يعني كل رمز ليه تمثيل Hexadecimal يبدأ بـ `%`.

وده بيساعد الرمز يوصل للسيرفر بدون ما يتغير أو يتمنع في بعض الحالات.

## آخر نقطة في الدرس

الكتاب بيقول:

سواء الموقع مكتوب بـ:

- PHP
    
- NodeJS
    
- ASP.NET
    
- Python
    
- Java
    

**مش فارقة.**

ليه؟

لأن اللي في الآخر بينفذ الأمر هو **نظام التشغيل (Shell)**، مش لغة البرمجة نفسها.

لغة البرمجة مجرد وسيط بيقول للـ OS:

> "نفذ الأمر ده."

فطالما المطور مرر مدخلات المستخدم للـ Shell من غير حماية، فطريقة اكتشاف واستغلال الـ Command Injection هتكون متشابهة جدًا بغض النظر عن لغة البرمجة المستخدمة.

**وده السبب إن فهمك للـ Shell Operators أهم بكتير من حفظ دوال PHP أو NodeJS.** لما تفهم إزاي الـ Shell بيفسر الأوامر، هتقدر تتعامل مع Command Injection على أغلب التقنيات المختلفة.

#$ آخر نقطة في الدرس

الكتاب بيقول:

إحنا لحد دلوقتي بنتعامل مع **Direct Command Injection**.

يعني:

```
Input
    ↓
System Command
    ↓
Output بيرجعلك
```

وده أسهل نوع.

لكن فيه أنواع أصعب هتدرسها بعدين، زي:

- **Blind Command Injection**: الأمر بيتنفذ لكن مفيش Output ظاهر، فتضطر تستنتج نجاحه بطرق غير مباشرة (زي الوقت أو الـ DNS أو الـ HTTP requests).
    
- **Indirect Command Injection**: مدخلاتك متتنفذش فورًا، لكنها توصل في مرحلة تانية لجزء بينفذ أوامر.
    

فابدأ بإتقان الـ **Direct Command Injection** الأول، لأنه الأساس اللي هتفهم منه كل الأنواع المتقدمة بعد كده.

## ليه كل الطرق دي شغالة؟

لاحظ إن كلهم مبنيين على نفس الفكرة.

الفلتر بيشتغل على **النص اللي إنت بعته**.

أما الـ Shell بيشتغل على **النص بعد ما يفسره**.

يعني فيه مرحلتين:

```
User Input
        ↓
Filter يشوف النص
        ↓
Shell يفسر النص
        ↓
Execution
```

المهاجم بيحاول يكتب حاجة:

- الفلتر يشوفها بشكل.
    
- لكن الـ Shell يفسرها بشكل مختلف.
    

وده جوهر معظم تقنيات الـ Filter Bypass.

## ليه الـ Blacklist ضعيف؟

لأن المطور لو قال:

```php
امنع Space
```

أنت عندك:

- Tab
    
- `${IFS}`
    
- Brace Expansion
    
- وطرق تانية كتير حسب الـ Shell والبيئة.
    

يعني كل مرة يمنع طريقة، ممكن تلاقي طريقة تانية تؤدي نفس الغرض.

وده السبب إن الاعتماد على **Blacklists** لوحدها بيكون ضعيف، بينما الحماية الأقوى هي **عدم تمرير مدخلات المستخدم إلى أوامر النظام أصلًا** أو استخدام **Allowlist (Whitelist)** والتحقق الصارم من المدخلات، مع تجنب استدعاء الـ Shell إذا كان فيه بديل آمن.
